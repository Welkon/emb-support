# TBS1102B notes

## Scope family

`TBS1102B` belongs to the `TBS1000B` family, so the official `Digital Oscilloscope Series Programmer Manual` for `TBS1000B` applies to it.

## What works well for this skill

- Remote control over the rear USB Device port through USBTMC.
- Direct SCPI queries and writes.
- Structured `control list/show/get/set/action` over a catalog of TBS1000B-family SCPI command groups.
- Numeric waveform capture with `CURVE?` plus `WFMOutpre` or `WFMPRE` scaling fields.

## Recommended connection workflow

1. Install Tek PC communications support or a VISA stack on the PC before connecting the rear USB Device port.
2. Install `pyvisa`.
3. Install `pyvisa-py` if no NI-VISA backend is present.
4. Run:

```bash
python3 scripts/tek_scope.py scan --backend @py
```

5. Find the `USB0::...::INSTR` resource whose `*IDN?` contains `TEKTRONIX,TBS1102B`.
6. Reuse that resource string for `idn`, `query`, `write`, and `waveform`.

If the user just wants a quick `CH1/CH2` dump, skip the manual resource step and run:

```bash
python3 scripts/tek_scope.py capture --backend @py --outdir captures --prefix tbs1102b
```

The helper auto-detects one matching Tek USB device. If multiple Tek USB scopes are attached, it stops and asks for an explicit `--resource`.

If the user wants full remote control instead of only waveform export, inspect the structured SCPI catalog first:

```bash
python3 scripts/tek_scope.py control list
python3 scripts/tek_scope.py control list --group vertical
python3 scripts/tek_scope.py control show trigger.main.edge.source
```

The default `control` view is now filtered to the TBS1102B subset that passed live validation.
To inspect family-level entries that are unsupported on this model or intentionally guarded because they can recalibrate, reset, recall, save, or alter external storage, add `--all`:

```bash
python3 scripts/tek_scope.py control list --all
python3 scripts/tek_scope.py control show --all system.autoset
```

Typical examples:

```bash
python3 scripts/tek_scope.py control get \
  --resource USB0::0x0699::0x0368::C000000::INSTR \
  --backend @py \
  --channel CH1 \
  channel.scale

python3 scripts/tek_scope.py control set \
  --resource USB0::0x0699::0x0368::C000000::INSTR \
  --backend @py \
  --channel CH1 \
  channel.scale \
  --value 0.5

python3 scripts/tek_scope.py control set \
  --resource USB0::0x0699::0x0368::C000000::INSTR \
  --backend @py \
  trigger.main.edge.source \
  --value CH1

python3 scripts/tek_scope.py control action \
  --resource USB0::0x0699::0x0368::C000000::INSTR \
  --backend @py \
  trigger.force
```

Guarded operations require explicit opt-in:

```bash
python3 scripts/tek_scope.py control action \
  --resource USB0::0x0699::0x0368::C000000::INSTR \
  --backend @py \
  --allow-risky \
  system.autoset
```

The `control` catalog covers the major TBS1102B command families from the official programmer manual: calibration/diagnostic, counter, cursor, data logging, display, FFT, filesystem, hardcopy, horizontal, limit test, math, measurement, system, PictBridge, save/recall, status, trend plot, trigger, vertical, and waveform transfer.
For anything not yet convenient to express as a catalog key, keep using raw `query` and `write`; they remain the exact full-coverage path for arbitrary SCPI strings.
Live validation on April 22, 2026 confirmed `196` working query paths, `117` working write-back paths, and `10` safe action paths on this specific `TBS1102B`; unsupported and guarded paths are now hidden from the default catalog profile. The verification artifacts are:

- `captures/tbs1102b-control-full-hw-test-20260422-r2.json`
- `captures/tbs1102b-control-failed-recheck-20260422.json`

## Intermittent-event monitoring

`TBS1102B` is practical for burst or intermittent fault monitoring, but the reliable path is to let the scope hardware catch the event first, then export waveform evidence. The helper now includes a `monitor` command for this workflow.

Recommended pattern for one-shot or burst anomalies:

```bash
python3 scripts/tek_scope.py monitor \
  --resource USB0::0x0699::0x0368::C000000::INSTR \
  --backend @py \
  --mode trigger \
  --match SAVE \
  --setup-channel CH1 \
  --channel-scale 1.0 \
  --channel-position 0 \
  --horizontal-scale 1.0E-4 \
  --trigger-type pulse \
  --trigger-source CH1 \
  --trigger-level 1.0 \
  --trigger-mode NORMal \
  --pulse-when LESSthan \
  --pulse-width 5.0E-6 \
  --pulse-polarity POSitive \
  --arm-stopafter SEQuence \
  --auto-rearm \
  --channels CH1 \
  --max-events 5 \
  --outdir captures \
  --prefix burst-watch
```

The helper now also includes built-in presets so the agent can choose a fault pattern first and then apply only the product-specific thresholds:

```bash
python3 scripts/tek_scope.py monitor --list-presets
python3 scripts/tek_scope.py monitor --show-preset glitch-short
```

Common preset-driven templates:

```bash
# Short glitch / narrow pulse
python3 scripts/tek_scope.py monitor \
  --resource USB0::0x0699::0x0368::C000000::INSTR \
  --backend @py \
  --preset glitch-short \
  --setup-channel CH1 \
  --trigger-source CH1 \
  --trigger-level 0.8 \
  --channel-scale 0.5 \
  --channel-position 0 \
  --channels CH1 \
  --outdir captures \
  --prefix glitch-watch

# Power-on spike
python3 scripts/tek_scope.py monitor \
  --resource USB0::0x0699::0x0368::C000000::INSTR \
  --backend @py \
  --preset power-on-spike \
  --setup-channel CH1 \
  --trigger-source CH1 \
  --trigger-level 1.0 \
  --channel-scale 2.0 \
  --channels CH1 \
  --max-events 1 \
  --outdir captures \
  --prefix power-on-watch

# Missing pulse / stretched pulse
python3 scripts/tek_scope.py monitor \
  --resource USB0::0x0699::0x0368::C000000::INSTR \
  --backend @py \
  --preset missing-pulse \
  --setup-channel CH1 \
  --trigger-source CH1 \
  --trigger-level 1.2 \
  --channel-scale 1.0 \
  --pulse-width 8.0E-5 \
  --channels CH1 \
  --outdir captures \
  --prefix missing-pulse-watch
```

If the bench setup needs to be reused, save the tuned plan once and reload it later:

```bash
python3 scripts/tek_scope.py monitor \
  --preset burst-pulse \
  --trigger-source CH1 \
  --trigger-level 1.0 \
  --channel-scale 1.0 \
  --save-profile captures/burst-pulse-profile.json \
  --dry-run

python3 scripts/tek_scope.py monitor \
  --profile captures/burst-pulse-profile.json \
  --resource USB0::0x0699::0x0368::C000000::INSTR \
  --backend @py
```

Use `trigger` mode when the scope has already been configured to trigger on the fault.
Use `limit` mode when the scope limit-test engine is the event detector.
Use `query` mode only when the user already knows a stable SCPI status query that flips when the event happens.

Each detected event writes:

- `<prefix>-eventNNN-<channel>.csv`
- `<prefix>-eventNNN-<channel>.json`
- `<prefix>-eventNNN-<channel>-report.txt`
- `<prefix>-events.json`

The new command is still polling scope state over USBTMC; it is not a true live sample stream into AI. For fast glitches, the scope trigger configuration remains the key part of the solution.
This means the practical workflow is now:

1. agent sends one `monitor` command that writes the intended scale, position, timebase, trigger, and arm settings
2. human reproduces the product anomaly
3. scope catches the event
4. helper exports waveform evidence automatically

`monitor` now also verifies the arm state after the initial `SEQuence + RUN` write and after every auto-rearm. The result JSON includes `arm_verification`, which captures the readback from `ACQuire:STOPAfter?`, `TRIGger:STATE?`, and `ACQuire:STATE?`. On April 22, 2026, live verification on this exact `TBS1102B` confirmed that auto-rearm could read back `STOPAfter=SEQUENCE` after a captured event.

## Waveform caveat

The programmer manual notes that if the waveform selected by `DATA:SOUrce` is not displayed, `CURVE?` returns nothing and raises an execution error. On this model, check display state first if capture unexpectedly comes back empty.
The new `capture` command handles this conservatively by default: hidden channels are reported as skipped instead of being modified. Use `--show-hidden` only when the user has accepted a display-state change.
During live validation on a `TBS1102B`, the working preamble family was `WFMPre:*` such as `WFMPre:XINcr?` and `WFMPre:YMUlt?`. Treat that family as the first choice on this scope before trying `WFMOutpre:*`.
The skill also computes a lightweight `analysis` summary after capture, including `v_min`, `v_max`, `v_pp`, `v_avg`, `v_rms`, estimated high/low levels, threshold-midpoint edge counts, estimated frequency, estimated duty cycle, and average high/low pulse widths.
If a user wants a short readable result instead of JSON, run `report --metadata <capture>.json`.

## Screenshot caveat

`TBS1102B` can save screen images, but the documented workflows center on:

- front-panel save to a USB flash drive
- hardcopy behavior over the USB path
- Tek PC communication software

This skill therefore treats waveform CSV capture as the primary AI-ingest path. It is more portable and easier to automate robustly than model-specific screenshot transfer.

## WSL USB permissions

When the scope is attached into WSL with `usbipd`, the USB device node may appear as `root:root` and deny access to a normal user. During live validation, the attached scope appeared as:

- vendor: `0699`
- product: `0368`

The non-persistent fix is:

```bash
sudo chgrp plugdev /dev/bus/usb/<bus>/<device>
sudo chmod 660 /dev/bus/usb/<bus>/<device>
```

For a persistent fix inside WSL, install a udev rule:

```bash
sudo bash scripts/install_wsl_usb_rule.sh
```

That script installs:

```udev
SUBSYSTEM=="usb", ATTR{idVendor}=="0699", ATTR{idProduct}=="0368", MODE="0660", GROUP="plugdev"
```

Then reattach the scope with `usbipd attach --wsl ...` or unplug and reconnect it so the new permissions take effect.
